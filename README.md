# Ramp-System: 就労移行支援業務統合システム

現場の記録、運営のコンプライアンス、経営の意思決定を統合し、支援の質と業務効率を最大化する「就労移行支援」特化型SaaS。

## 🚀 主要機能
- **リアルタイム活動トラッカー**: ダッシュボードから直接支援・間接業務をワンクリックで記録。
- **インテリジェント日報連携**: 記録した活動を自動的に日報タイムラインへ反映し、転記ミスをゼロに。
- **個別支援計画連動**: すべての支援記録を計画目標と紐付け、算定根拠と監査証跡を自動生成。
- **日本時間 (JST) 最適化**: サーバー所在地を問わず、日本のカレンダーと時刻に基づいた業務運用を実現。

## 🛠 技術スタック

### バックエンド (Python 3.12+)
- **フレームワーク:** Flask (Web API)
- **データベース操作 (ORM):** SQLAlchemy / Flask-SQLAlchemy
- **マイグレーション:** Flask-Migrate (Alembic)
- **認証・セキュリティ:** Flask-JWT-Extended (Cookie-based JWT with CSRF Protection)
- **暗号化:** Flask-Bcrypt, Cryptography (PII暗号化)
- **テスト:** pytest

### フロントエンド (Modern Web Stack)
- **フレームワーク:** React 18 / Vite
- **言語:** TypeScript (Strict Mode)
- **スタイリング:** Tailwind CSS
- **アイコン:** Lucide React
- **通信:** Axios (CSRF Token Auto-injection)

### インフラ・DB
- **RDBMS:** PostgreSQL / SQLite (Local/Test)
- **タイムゾーン:** Asia/Tokyo (JST)

  - **計画承認時における本人同席会議の必須化と例外対応**: 計画原案（DRAFT）の承認プロセスにおいて、本人参加（`user_participated=True`）の会議記録の存在を必須条件化。本人が不在となるやむを得ない場合には、サービス管理責任者による「デジタル宣誓（デジタル承認）」および不在時モニタリング概要・不在理由の入力を求めるガードレールを構築。

- **本稼働（プロダクション）に向けた致命的な欠陥の解消とアーキテクチャ改修 (2026年5月)**:
  - **Fat Modelの解消 (TypeDecorator導入)**: データベースモデル (`UserPII`, `SupporterPII`) 内に散在していた暗号化/復号のプロパティを全廃し、SQLAlchemyの `TypeDecorator` を用いたカスタム型 `EncryptedString` に責務をカプセル化。これによりバルク処理時のパフォーマンス（N+1問題）が解消され、モデルが単なるデータ構造定義に専念できるようになりました。
  - **Fat Controllerの解消 (AuthService分離)**: `auth.py` (Controller) 内に混在していた職員・利用者の認証分岐やJWTクレーム生成のビジネスロジックを `auth_service.py` へ抽出し、単一責任の原則 (SRP) を徹底しました。
  - **セッション復元の自動化とAPIセキュア化**: ページリロード時に強制ログアウトされる問題を解決。`/me` エンドポイントでの無効IDフォールバックを排除（`401`エラー返却）し、より厳格なセッション認証を実装。
  - **フロントエンドの責務・型安全性の浄化**: `apiClient.ts` のAxiosインターセプターで手動実装していたCSRF付与処理を削除し、Axios標準の `xsrfCookieName` 設定に移行。また、`AuthContext.tsx` 内の `catch (err: any)` を `catch (err: unknown)` と `axios.isAxiosError` に置き換え、TypeScriptの型安全性を向上させました。
  - **CSRF対策と鍵のフェイルセーフ**: 本番環境でのCSRF保護を強制有効化し、必須暗号化鍵が欠落している場合には直ちにシステムを停止させる強固な安全措置を導入。
