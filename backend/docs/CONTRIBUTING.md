# 🏗️ RAMP System 開発・貢献ガイドライン  
このドキュメントは、バージョン管理とGitコミットのルールを定義し、チーム全員（および未来の自分）が迷いなく開発を進めるための指針です。  
## 🏷️ 1. バージョン管理の定義 (Versioning Strategy)
本プロジェクトでは、**セマンティック バージョニング (Semantic Versioning 2.0.0)** に基づき、以下のルールでバージョンを管理します。  

`vX.Y.Z` (例: v2.3.1)  
| 項目 | 意味 |インクリメントのタイミング |
| :---- | :---- | :---- |
| X (Major) | メジャー設計思想の根本的な変更や、互換性のない大規模な刷新。 | （例：DB構造の完全リセット、アーキテクチャ変更）|
| Y (Minor) | マイナー新しい機能の追加。 | （例：チャット機能の実装、新しいAPIの追加）※後方互換性は維持する。 |
|Z (Patch)|パッチバグ修正や、軽微な調整。|（例：誤字修正、計算ロジックの微修正）|

## 💡 現在のステータス（2025-11-22時点）
- `v2.3`: DB土台の完全構築完了版（マスタ投入済み）。
  - 次に「サービス層の実装」が完了したら `v2.4` へ。
  - API層が完成したら `v2.5` へ。
  - フロントエンドと結合してリリースする際に `v3.0` とする予定。

## 💬 2. コミットメッセージのルール (Commit Message Convention)
コミットログを見れば「何をしたか」が一瞬で分かるよう、以下の**プレフィックス（接頭辞）**を必ず付けてください。  
### 📝 フォーマット
`[Prefix]: [変更内容の要約]`  
例:` Feat: 職員情報の暗号化ロジックを追加`  
🔑 プレフィックス一覧  
| Prefix | 意味 | 使用例 |
| :--- | :--- | :--- |
| `Feat` | 新機能 (Feature) | `Feat: チャット機能の実装` |
| `Fix` | バグ修正 | `Fix: 常勤換算の計算ミスを修正` |
| `Docs` | ドキュメント | `Docs: READMEに技術スタックを追記` |
| `Refactor` | リファクタリング | `Refactor: 変数名を整理（機能変更なし）` |
| `Test` | テストコード | `Test: ログイン認証のテストを追加` |
|`Chore` | 雑務・設定 | `Chore: .gitignoreの更新、Chore: パッケージのインストール` |
| `WIP` |作業中 (Work In Progress) | `WIP: サービス層の実装（途中）` ※基本は避けるが、一時保存用 |

## 🌿 3. ブランチ戦略 (Branching Strategy) - 推奨
一人開発であっても、機能ごとにブランチを切ることで「安全」に開発できます。
- main: 常に動作する、安定した最新版。（直接コミットは避ける）
- develop: 開発中のメインブランチ。
- feature/機能名: 新機能開発用。
   - （例: feature/auth-service）作業完了後、develop にマージする。

### 🚀 基本的なワークフロー
1. 作業開始: `git checkout -b feature/新機能名`
2. 実装 & テスト: コードを書き、`pytest` で確認。
3. コミット: `git add .` → `git commit -m "Feat: 〇〇機能を追加"`
4. マージ: `main` (または `develop`) に戻り、`git merge feature/新機能名`
5. プッシュ: `git push origin main`
